# Архитектура и технологии проекта

## Общая архитектура

Проект построен по модульной архитектуре с четким разделением ответственности между компонентами. Архитектура позволяет легко масштабировать и поддерживать систему.

```
┌─────────────────────────────────────────────────────────────┐
│                    Пользовательский уровень                  │
├─────────────────────────────────────────────────────────────┤
│  Telegram Bot  │  Mini App Frontend  │  Admin Panel        │
├─────────────────────────────────────────────────────────────┤
│                    API уровень                               │
├─────────────────────────────────────────────────────────────┤
│  Express Server  │  Bot Framework  │  Static Files         │
├─────────────────────────────────────────────────────────────┤
│                    Данные уровень                            │
├─────────────────────────────────────────────────────────────┤
│  In-Memory DB  │  File System  │  External APIs           │
└─────────────────────────────────────────────────────────────┘
```

## Технологический стек

### Backend (Node.js)

#### Основные технологии
- **Node.js** - JavaScript runtime
- **Express.js** - Web framework
- **Telegraf** - Telegram Bot Framework
- **Body Parser** - Request parsing middleware

#### Архитектурные решения
- **RESTful API** - Для взаимодействия с frontend
- **Middleware pattern** - Для обработки запросов
- **Error handling** - Централизованная обработка ошибок
- **Environment variables** - Конфигурация через .env

#### Структура backend
```
server.js          # Основной сервер
bot.js            # Telegram бот
├── API endpoints  # REST API маршруты
├── Middleware     # Промежуточная обработка
└── Error handling # Обработка ошибок
```

### Frontend (Client-side)

#### Основные технологии
- **HTML5** - Структура страниц
- **CSS3** - Стилизация и анимации
- **JavaScript (ES6+)** - Логика приложения
- **Telegram Web Apps SDK** - Интеграция с Telegram

#### Архитектурные решения
- **Component-based** - Модульная структура
- **Event-driven** - Обработка пользовательских действий
- **SPA (Single Page Application)** - Одностраничное приложение
- **Responsive design** - Адаптивный дизайн

#### Структура frontend
```
public/
├── index.html     # Главная страница
├── styles.css     # Стили приложения
├── app.js         # JavaScript логика
└── assets/        # Ресурсы (изображения, иконки)
```

## Архитектурные паттерны

### 1. MVC (Model-View-Controller)

**Model** (Данные):
- Контакты пользователей
- Категории бизнеса
- Состояние приложения

**View** (Представление):
- HTML шаблоны
- CSS стили
- Визуальные компоненты

**Controller** (Контроллер):
- Обработка пользовательских действий
- Валидация данных
- Управление состоянием

### 2. Observer Pattern

**Использование**:
- Отслеживание изменений состояния
- Обновление UI при изменениях данных
- Обработка пользовательских событий

**Пример**:
```javascript
// Наблюдение за изменениями в корзине
function updateCartDisplay() {
    // Обновление отображения корзины
}

// Подписка на изменения
cart.subscribe(updateCartDisplay);
```

### 3. Factory Pattern

**Использование**:
- Создание шаблонов мини-приложений
- Генерация компонентов
- Управление зависимостями

**Пример**:
```javascript
function createTemplate(category) {
    switch(category) {
        case 'restaurant': return renderRestaurantTemplate();
        case 'retail': return renderRetailTemplate();
        // ...
    }
}
```

## База данных

### Текущая реализация (In-Memory)

**Хранение данных**:
- В оперативной памяти сервера
- Подходит для демо-версии
- Простота реализации

**Структура данных**:
```javascript
const contacts = [
    {
        id: 'timestamp',
        category: 'restaurant',
        phone: '+79991234567',
        timestamp: 'ISO date',
        userAgent: 'browser info'
    }
];
```

### Планы на production

**PostgreSQL**:
- Реляционная база данных
- Надежность и производительность
- Поддержка сложных запросов

**MongoDB**:
- Документоориентированная база
- Гибкость схемы
- Хорошо подходит для JSON-данных

**Redis**:
- Кэширование данных
- Сессии пользователей
- Очереди задач

## API Архитектура

### RESTful принципы

**Ресурсы**:
- `/api/contacts` - Контакты пользователей
- `/api/contacts/category/:category` - Контакты по категориям
- `/health` - Проверка состояния сервера

**HTTP методы**:
- `GET` - Получение данных
- `POST` - Создание данных
- `DELETE` - Удаление данных

**Статусы ответов**:
- `200` - Успешный запрос
- `201` - Ресурс создан
- `400` - Ошибочный запрос
- `404` - Ресурс не найден
- `500` - Внутренняя ошибка сервера

### Примеры API запросов

**Создание контакта**:
```http
POST /api/contacts
Content-Type: application/json

{
    "category": "restaurant",
    "phone": "+79991234567"
}
```

**Получение контактов**:
```http
GET /api/contacts
```

**Получение по категории**:
```http
GET /api/contacts/category/restaurant
```

## Безопасность

### 1. Валидация данных

**На стороне клиента**:
- Проверка форматов телефонов
- Валидация обязательных полей
- Ограничение длины вводимых данных

**На стороне сервера**:
- Проверка типов данных
- Санитизация входящих данных
- Защита от SQL-инъекций

### 2. CORS (Cross-Origin Resource Sharing)

**Настройки**:
- Разрешение запросов с доверенных доменов
- Ограничение методов
- Контроль заголовков

### 3. HTTPS

**Требования**:
- Обязательное использование HTTPS в production
- SSL/TLS сертификаты
- Защита передаваемых данных

### 4. Rate Limiting

**Ограничения**:
- Количество запросов с одного IP
- Защита от DDoS атак
- Контроль нагрузки на сервер
